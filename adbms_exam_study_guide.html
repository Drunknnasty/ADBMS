ADBMS Exam Study Guide — Interactive review of deadlock detection, recovery, fragmentation, and distributed databases

Week 6 — Deadlock Detection & Recovery
What is a deadlock?
The mental model you need before anything else
Definition
An impasse where two or more transactions each wait for locks held by the other — creating an infinite wait.
Real-world analogy
Two cars at a narrow one-lane bridge, each waiting for the other to reverse. Neither can proceed.
Why it matters on your exam
Exams give you a schedule and ask: "Does a deadlock exist?" You need to build a WFG and detect cycles.
⚠️ Common mistake: Students confuse deadlock prevention (wait-die, wound-wait, timeout) with deadlock detection (WFG). Prevention avoids deadlocks before they happen. Detection checks if one already exists. Your exam is about detection.
Wait-for Graph (WFG) — the 3-step algorithm
Memorize this sequence exactly — exams follow it step by step
Step 1
Create one node per transaction. (T1, T2, T3…)
Step 2
For each transaction, check every Lock() call. If the item is already locked by another transaction, draw an arrow Ti → Tj.
Step 3
Check for a cycle. Cycle = deadlock. No cycle = no deadlock.
The arrow direction means: "Ti is waiting FOR Tj". So T1→T2 means "T1 is blocked waiting for T2 to release a lock."
Interactive WFG Example
Click a node to see what it's waiting for
Recovery from deadlock — the 3 factors
Once a deadlock is detected, one transaction must be killed (rolled back)
Factor 1 — Choice of victim
Kill the "cheapest" transaction: younger (ran less time), fewer data items updated, fewer data items still to update. Always prefer less damage.
Factor 2 — How far to rollback
Simplest: full rollback (undo everything). More efficient: partial rollback (undo only enough to break the deadlock).
Factor 3 — Starvation
Never keep killing the same transaction. DBMS tracks a "victim count" — once a threshold is hit, it must pick someone else.
⚠️ Starvation exam trap: Students often forget starvation. The fix is timestamping — when a transaction is restarted, it keeps its ORIGINAL timestamp so it ages and gains priority over time. Never assign a new timestamp on restart.
Week 7 — Database Recovery: Checkpoint, Update Methods, DBMS Actions
The big picture: why recovery is needed
Understanding the problem makes the solutions obvious
The problem
Writing to a database is NOT instant. Data sits in a RAM buffer first, then gets flushed to disk. A crash between these two steps causes data loss.
Real-world analogy
You type a document (RAM), then hit Save (disk). If your computer crashes before Save — the work is gone. Recovery is the "auto-save" system.
The recovery manager's job
Guarantee ATOMICITY (all or nothing) and DURABILITY (committed data survives crashes). It does this using logs, checkpoints, rollback and rollforward.
Checkpoint — the game save point
The exact analogy: a video game checkpoint

A checkpoint forces all in-memory (RAM) data to be written permanently to disk. Without checkpoints, the system would have to replay all transactions from the very beginning after a crash. With checkpoints, it only replays from the last checkpoint.

Checkpoint operations: (1) Write all log records from memory to disk, (2) Write all modified database buffers to disk, (3) Write a checkpoint record to the log (listing all active transactions at that moment).
⚠️ Checkpoints are ONLY relevant in Deferred Update mode. In Immediate Update, data is always on disk already.
Immediate vs Deferred Update — the most exam-tested concept
Students who confuse these lose the most marks
Immediate UpdateDeferred Update
When data goes to diskAs soon as WRITE is issuedOnly when a CHECKPOINT appears
SpeedSlower (frequent disk writes)Faster (batch writes at checkpoint)
Data loss riskLess (data is always on disk)More (data may be in RAM only)
The lorry analogyLorry takes one item, goes, comes backLorry waits for ALL items, then drives
🚨 The critical difference: In Deferred Update, a transaction can COMMIT but its data is still only in RAM (not on disk yet). This is the scenario that causes ROLLFORWARD.
DBMS Actions on Failure — the decision table
This is the most direct exam question type. Memorize the logic, not the table.
Logic: Data is ALWAYS on disk when written. So ask: "Was the transaction still running at failure time?"
Situation at failureActionWhy
Transaction not yet startedNothingTransaction had no effect on database
Transaction running (started, not committed)RollbackData was written to disk, but transaction didn't finish — must undo partial changes
Transaction completed (committed)NothingData is on disk and transaction is done — nothing to do
Logic: Data only goes to disk at checkpoints. Ask: "Was there a checkpoint AFTER the commit? Was the transaction still running at checkpoint?"
Situation at failureActionWhy
Transaction not yet startedNothingNo effect at all
Running, NO checkpoint occurred yetNothingData never reached disk — database is unchanged from before transaction started
Committed, but NO checkpoint after commitRollforwardTransaction succeeded but data is still in RAM (lost in crash). Must redo all operations to recover committed data.
Running, checkpoint occurred during executionRollbackPARTIAL data made it to disk (at checkpoint), but transaction never committed — must undo what was written
Committed AND checkpoint occurred after commitNothingData is safely on disk, transaction complete
⚠️ Rollforward (also called Redo/Rollforward) means: re-execute all the transaction's operations. It happens ONLY in deferred update when: transaction committed BUT no checkpoint appeared after that commit before the crash.
Recovery facilities a DBMS must provide
Expect a short-answer question listing these
Backup mechanism
Periodic full or incremental copies of the entire database, stored on offline media (tape).
Log file (journal)
Records every operation: transaction ID, operation type, object affected, before-image, after-image, and pointers.
Checkpoint + Recovery manager
Checkpoint forces RAM→disk sync. Recovery manager restores consistent state after failure using the log.
Week 8 — Centralized vs Distributed DB, Data Delivery Alternatives
DDB, D-DBMS, and DDBS — three terms students mix up
Learn the hierarchy: DDBS = DDB + D-DBMS
DDB — the data
Multiple logically interrelated databases distributed over a network. Just the data — no management software.
D-DBMS — the software
The software that manages the DDB and makes distribution transparent to users (e.g., Oracle, SQL Server, MySQL, MongoDB).
DDBS — the complete system
DDB + D-DBMS together. This is the full distributed database system. Formula: DDBS = DDB + D-DBMS
⚠️ What is NOT a DDBS: A timesharing system (one machine, multiple users), a tightly-coupled multiprocessor (shared memory), or a centralized database on a network node (one DB, many clients).

Week 9–10 — Fragmentation: Horizontal, Vertical, Correctness Rules
What is fragmentation and why do it?
The motivation makes the types make sense
Definition
Dividing a relation (table) into smaller pieces called fragments, each stored at a different site in the distributed database.
Analogy
An engineering firm with offices in Lahore, Islamabad, and Karachi stores each city's employee data locally — not everything at one central server.
Why not just replicate?
Replication stores the full table everywhere (high memory, every update runs at all sites). Fragmentation is selective — store only what's needed locally.
3 Correctness Rules — the foundation of every fragmentation question
These rules are always tested. Understand WHY they exist.
Completeness
Every record in the original table must belong to at least one fragment. No record can be "lost."

Test: Can I find every original row in some fragment?
Reconstruction
No fragment should accept a record that is INVALID for the original table. Fragments must respect the table's constraints.

Test: Would an invalid record sneak into any fragment?
Disjointness
Each record must belong to ONE and only ONE fragment. No record can appear in two fragments simultaneously.

Test: Does any record appear in more than one fragment?
🚨 Most common rule violated: Disjointness. When you use a condition like City != 'ISB', records that are 'ISB' can slip into another fragment that uses City = 'ISB'. Always check for overlapping conditions.
Horizontal Fragmentation — splitting by rows
Two types: Primary and Derived

Primary: Fragment a relation using predicates defined ON THAT SAME RELATION.

When fragments are based on multiple columns, using simple conditions causes Disjointness violations. The fix: use MINTERMS — combinations of predicates that are mutually exclusive.

Minterm procedure (multi-column fragmentation):

StepAction
1List all predicates (P1, P2, P3, P4…) from the application queries
2Create minterms = combinations of predicates (P1∧P2, P1∧P3, P2∧P3…)
3Discard minterms that have 0 records OR are duplicates (P1∧P2 = P2∧P1)
4Each remaining valid minterm becomes a fragment
5Stop when all original records are covered by some minterm
⚠️ P1∧P2 and P2∧P1 are THE SAME minterm. Once you evaluate one, skip the other. This is a common source of wasted effort in exams.

Derived: Fragment relation R based on predicates defined on a DIFFERENT relation S (usually a related table via foreign key).

Procedure: (1) Perform Primary Horizontal Fragmentation on relation S to get minterms. (2) Semi-join each minterm of S with relation R to get R's fragments.

Semi-join notation: Project ▷ m2 means: "Keep only the Project rows whose EmpNo appears in minterm m2 of Employee."

Memory tip: Derived = "derived FROM another table's fragmentation." Employee is fragmented first → Project is fragmented BASED ON those Employee fragments.

Vertical Fragmentation: Split a table by COLUMNS, not rows. Each fragment contains a subset of columns + the primary key (always).

Why include primary key?
Without the primary key, you can't reconstruct (join) the original table back. The PK is the link between all vertical fragments.
Affinity Matrix
Shows how often pairs of columns appear together in queries. High affinity = put those columns in the same fragment. Build by summing query frequencies.
Frequency step
Sum each query's execution count across all sites to get total frequency. Then, for each column pair, sum frequencies of queries that use BOTH columns.
⚠️ Vertical fragmentation does NOT need to satisfy Disjointness for columns — the same column (especially PK) appears in multiple fragments. Disjointness only applies strictly to HORIZONTAL fragmentation (rows).
