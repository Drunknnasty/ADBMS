Week 6 — Deadlock Detection & Recovery
What is a deadlock?
The mental model you need before anything else
Definition
Two or more transactions each waiting for locks held by the other — an infinite stall with no way out.
Real-world analogy
Two cars at a narrow one-lane bridge, each waiting for the other to reverse. Neither can move.
Exam relevance
Given a schedule, you must build a WFG and declare: deadlock or no deadlock? That's the core question.
Common mistake: confusing deadlock prevention (wait-die, wound-wait) with deadlock detection (WFG). Your exam is about detection — checking if a deadlock already exists in a given schedule.
Wait-for Graph — the 3-step algorithm
Memorize this sequence. Exams follow it exactly.
Step 1
Create one node for each transaction. (T1, T2, T3…)
Step 2
For each Lock() call, if that item is already locked by Tj, draw an arrow Ti → Tj.
Step 3
Detect a cycle → Deadlock. No cycle → No deadlock.
Arrow direction means: Ti is waiting FOR Tj. So T1→T2 means "T1 is blocked waiting for T2 to release its lock."
Interactive WFG — click a node to inspect it
Left side: deadlock cycle. Right side: no cycle.
Recovery from deadlock — 3 factors
Once detected, one transaction must be killed (rolled back)
Factor 1 — Victim choice
Kill the "cheapest" transaction: younger (ran less time), fewer items already updated, fewer items still to update. Minimize damage.
Factor 2 — How far to rollback
Simplest: full rollback (undo everything). More efficient: partial rollback (undo just enough to break the deadlock).
Factor 3 — Avoid starvation
Never keep killing the same transaction. DBMS tracks a victim count per transaction. After a threshold, it must pick someone else.
Starvation fix = timestamping. On restart, the transaction keeps its ORIGINAL timestamp — it ages over time, gaining priority. Never assign a new timestamp on restart.
Week 7 — Checkpoint, Update Methods, DBMS Actions
Why recovery is needed — the big picture
Understanding the problem makes the solutions obvious
The problem
Data doesn't go directly from a transaction to disk. It sits in a RAM buffer first. A crash between buffer and disk = data loss.
Analogy
Typing a document (RAM) then hitting Save (disk). Crash before Save = work lost. Recovery is the auto-save system for your database.
Recovery manager's goal
Guarantee Atomicity (all-or-nothing) and Durability (committed data survives crashes) using logs, checkpoints, rollback, and rollforward.
Checkpoint — the game save point
Forces all RAM data to disk permanently

Without checkpoints, a crash requires replaying every transaction from the beginning. With checkpoints, only replay from the last checkpoint. Three things happen at a checkpoint:

1
All log records in memory → written to secondary storage (disk).
2
All modified database buffers → force-written to secondary storage.
3
A checkpoint record → written to the log (lists all active transactions at this moment).
Checkpoints are ONLY relevant in Deferred Update mode. In Immediate Update, data is already on disk so checkpoints don't change the recovery logic.
Immediate vs Deferred Update
The most exam-tested concept — students who confuse these lose the most marks
Immediate UpdateDeferred Update
When data hits diskAs soon as WRITE is issuedOnly when a CHECKPOINT appears
SpeedSlower (frequent disk writes)Faster (batch writes at checkpoint)
Data loss risk on crashLess — data is always on diskMore — data may be in RAM only
The lorry analogyLorry takes one item at a time, constantly movingLorry waits for ALL items, then drives once
Critical insight: In Deferred Update, a transaction can COMMIT but its data is still only in RAM. This exact scenario triggers ROLLFORWARD.
DBMS action at failure — the decision tables
The most direct exam question type. Learn the logic, not just the answer.
Logic: data is always on disk. Ask: "Was the transaction still running at failure time?"
Transaction status at failureActionWhy
Not started yetNothingNo effect on database at all
Running (not committed)RollbackPartial writes are on disk — must undo incomplete changes to restore consistency
Completed (committed)NothingData is on disk, transaction is done — nothing to change
Logic: data only hits disk at checkpoints. Ask: "Was there a checkpoint? Was it before or after the commit?"
Transaction status at failureActionWhy
Not started yetNothingNo effect at all
Running, no checkpoint yetNothingData never reached disk — database is unchanged from before transaction started
Committed, no checkpoint after commitRollforwardTransaction succeeded but data is in RAM only (wiped by crash). Must redo all operations to re-save to disk.
Running, checkpoint happened mid-transactionRollbackPartial data made it to disk at checkpoint, but transaction never committed — must undo partial writes
Committed AND checkpoint after commitNothingData is safely on disk, transaction fully complete
Rollforward = redo. Trigger: committed transaction + no checkpoint after commit. Re-executes all operations to save data permanently.
Week 8 — Centralized vs Distributed DB, Data Delivery Alternatives
DDB, D-DBMS, and DDBS — three terms students mix up
Learn the hierarchy first: DDBS = DDB + D-DBMS
DDB — the data
Multiple logically interrelated databases distributed over a network. The data only — no management software included.
D-DBMS — the software
Software that manages the DDB and makes distribution transparent to users. Examples: Oracle, SQL Server, MySQL, MongoDB, PostgreSQL.
DDBS — the full system
DDB + D-DBMS combined. The complete distributed database system. Formula: DDBS = DDB + D-DBMS
What is NOT a DDBS: (1) a timesharing system, (2) a tightly-coupled multiprocessor with shared memory, (3) a centralized database on a network node — one DB, many clients accessing it.

Week 9–10 — Fragmentation: Horizontal, Vertical, Correctness Rules
3 Correctness Rules — tested on every fragmentation question
Understand WHY each rule exists, not just its name
Completeness
Every record in the original table must belong to at least one fragment. No record can fall through the gaps.

Test: Can I find every row somewhere in a fragment?
Reconstruction
No fragment should accept a record that is INVALID for the original table. Fragments must respect the table's constraints.

Test: Would an invalid record sneak into any fragment?
Disjointness
Each record must belong to ONE and only ONE fragment. No duplicates across fragments.

Test: Does any record appear in more than one fragment?
Most common violation: Disjointness. Using City != 'ISB' as a condition overlaps with other fragments. Always check for overlapping conditions between fragments.
Horizontal, Derived & Vertical Fragmentation
Three types — know when each applies

Fragment a relation using predicates defined on that same relation. When fragments involve multiple columns, use minterms to avoid Disjointness violations.

StepAction
1List all predicates (P1, P2…) from the application queries — the WHERE clause conditions
2Create minterms = AND-combinations of predicates: P1∧P2, P1∧P3, P2∧P3…
3Discard minterms with 0 records, and skip duplicates (P1∧P2 = P2∧P1)
4Each remaining valid minterm becomes one fragment
5Stop when all original records are covered
P1∧P2 and P2∧P1 are the SAME minterm. Evaluate once, skip the reverse. A common time-waster in exams.

Fragment relation R using predicates defined on a different relation S (related via foreign key). Used when you want to co-locate related rows from child tables alongside the parent.

Procedure: (1) Perform Primary Horizontal Fragmentation on relation S to get minterms. (2) Semi-join each minterm with relation R. Each semi-join result = one derived fragment of R.
Semi-join notation
Project ▷ m2 means: keep only Project rows whose EmpNo appears in minterm m2 of Employee. Employee fragments → then Project fragments based on those.

Split a table by columns, not rows. Each fragment contains a subset of columns + the primary key (always mandatory).

Why include the PK?
Without the primary key, you can't join the fragments back together to reconstruct the original table. PK is the link.
Affinity matrix
Shows how often column pairs appear together in queries. High affinity = put those columns in the same fragment for efficiency.
How to build it
Sum each query's execution count across all sites = total frequency. For each column pair, sum frequencies of all queries that use BOTH columns.
Disjointness does NOT apply to columns in vertical fragmentation — the primary key legitimately appears in every fragment. Disjointness only strictly applies to rows in horizontal fragmentation.
